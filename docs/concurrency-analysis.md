# 并发数评估报告

## 场景分析

**总请求数**: 60,000次查询/天  
**查询频率**: 每日一次  
**约束条件**:
- GitHub Actions 超时限制: 6小时（21600秒）
- 目标服务器可能的限流
- Runner 性能限制

---

## 时间预算计算

### 方案1：保守估计（单次请求2秒）

```
总耗时 = 60,000 × 2秒 = 120,000秒 = 33.3小时 ❌ 超过6小时限制
```

### 方案2：使用并发

需要在6小时内完成：
```
最大允许耗时 = 6小时 = 21,600秒
最小并发数 = 60,000 × 2秒 / 21,600秒 = 5.56
```

**理论上最小并发数**: 约 **6个并发**

---

## 并发数优化分析

### 实际测量数据（基于现有脚本）

从 `nju_electric_query.py` 可以看到：
- 当前默认并发: 24
- 重试机制: 3次，指数退避
- 超时: 30秒/请求

### 推荐并发数计算

考虑以下因素：

1. **网络延迟**: 南京大学服务器响应时间
   - 国内访问: 100-500ms
   - 单次请求总耗时: 1-3秒（含解析）

2. **服务器限流风险**:
   - 并发过高可能触发限流
   - 建议: 逐步测试上限

3. **Runner 性能**:
   - Linux runner: 2核CPU, 8GB内存
   - 可支持 50-100 并发连接

### 推荐配置

| 并发数 | 预计耗时 | 风险评估 | 推荐度 |
|--------|---------|---------|--------|
| 10 | 2小时 | 低风险 | ⭐⭐⭐ |
| 20 | 1小时 | 中等风险 | ⭐⭐⭐⭐ |
| 30 | 40分钟 | 中等风险 | ⭐⭐⭐⭐⭐ **推荐** |
| 50 | 24分钟 | 高风险 | ⭐⭐ |
| 100 | 12分钟 | 很高风险 | ⭐ |

**推荐并发数**: **30-50**

## 详细计算（30并发）

假设：
- 单次请求平均耗时: 1.5秒
- 并发数: 30

计算：
```
批次数量 = 60,000 / 30 = 2,000批次
每批次耗时 = 1.5秒
总耗时 = 2,000 × 1.5秒 = 3,000秒 = 50分钟
```

**实际预留时间**: 考虑重试、网络波动，建议预留 1.5-2小时

---

## 风险评估

### 高并发风险

1. **服务器限流**
   - 症状: 403错误、连接超时
   - 解决: 降低并发数，增加重试间隔

2. **IP封禁**
   - 风险: GitHub Actions IP池共享
   - 缓解: 降低并发，分散请求时间

3. **内存溢出**
   - Runner限制: 8GB内存
   - 监控: 在workflow中添加内存检查

### 推荐策略

**渐进式测试**：
```python
# 第一天测试
并发数: 20
观察: 成功率、响应时间、错误类型

# 如果成功率高（>95%），第二天增加到30
并发数: 30

# 继续观察到稳定运行
并发数: 40-50（最大建议值）
```

## 实际配置建议

### Workflow 配置

```yaml
jobs:
  query-electricity:
    timeout-minutes: 120  # 2小时超时
    
    steps:
      - name: Query electricity data
        run: |
          python nju_electric_query.py \
            -d ./database \
            -c 30 \              # 并发数30
            --retry 3 \          # 重试3次
            --retry-delay 5 \    # 重试间隔5秒
            $(cat config/room_ids.txt)
```

### 分批查询策略（可选）

如果并发数受限，可以分批查询：

```yaml
jobs:
  query-batch-1:
    run: python nju_electric_query.py -c 20 room_ids_batch1.txt
  
  query-batch-2:
    needs: query-batch-1
    run: python nju_electric_query.py -c 20 room_ids_batch2.txt
  
  query-batch-3:
    needs: query-batch-2
    run: python nju_electric_query.py -c 20 room_ids_batch3.txt
```
