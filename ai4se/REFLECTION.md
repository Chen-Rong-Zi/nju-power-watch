- [x] 哪些 Superpowers 技能在你的项目里发挥了最大作用？哪些感觉是"形式大于实质"？

## 高价值技能

1. **brainstorming** — 在需求不明确时，能将模糊的初始想法系统地转化为成熟设计。视觉原型的输出帮助提前发现需求矛盾，降低了后期返工成本。

2. **systematic-debugging** — 使用频率最高的 skill，其结构化排查流程显著缩短了根因分析时间，避免了经验主义的盲目试错。

3. **test-driven-development** — 配合 requesting-code-review 构成了有效的质量闭环，在开发阶段拦截了大部分回归缺陷。

## 形式大于实质的技能

1. **writing-plans** — 实践中，plan 阶段往往已包含具体代码片段，导致后续实现退化为机械的"替换粘贴"任务。plan 的价值被上游 spec 和 brainstorming 阶段的细节程度挤压，边际收益递减。

- [x] TDD 强制在 AI 协作下的体感如何？它是阻碍还是放大器？

在单次 Superpowers 流程内，TDD 有效消除了显性缺陷，测试先行迫使 AI 明确行为边界。但在跨流程（如多轮迭代或多 agent 协作）场景中，TDD 反而引入摩擦：前一轮生成的测试可能在后一轮架构调整后失效，维护测试与生产代码的一致性成为额外开销。结论：TDD 是局部放大器、全局摩擦源，需要更精细的测试范围管理策略。

- [x] SPEC 与 PLAN 的质量如何影响了后续实现质量？是否有过"规约不清晰导致 subagent 跑偏"的具体案例？

1. **Spec 不清晰** → 实现偏差与返工。模糊的规约迫使 subagent 依赖隐含假设填补空白，最终输出与预期产生系统性偏离。典型案例：一次前端交互行为的 spec 未明确"空状态"和"错误状态"的区分，subagent 将二者合并处理，导致后续验收阶段才发现设计意图未被满足。

2. **Plan 不完整** → 架构缺陷与技术债务。缺少边界条件、错误路径或性能约束的 plan，在实现阶段暴露问题，修复成本呈指数级增长（验证了 Boehm 曲线）。具体行为：subagent 会选择"最短路"实现方案，而非"最鲁棒"方案。

核心结论：Spec 决定"做对的事"，Plan 决定"把事做对"。两者中的任何缺位都会使 subagent 的执行质量退化到纯模型先验的水平。

- [x] 你设计的最有效的 prompt / context 策略是什么？为什么有效？

前端开发场景中，先向 AI 精确描述正确的交互逻辑（而非直接要求生成代码），再让其通过 MCP 操作浏览器进行端到端验证。这一策略有效的根本原因在于：它将 AI 的注意力从"怎么写代码"引导到"怎么验证行为正确性"，通过自验证循环确保生成代码与需求语义一致，减少了对 prompt 措辞敏感性的依赖。

- [x] 你对 Superpowers 这套方法论的批判：它假设了什么？这些假设在你的项目里成立吗？

Superpowers 方法论包含若干未明言的假设，这些假设在实践中部分成立、部分存疑：

1. **可预知性假设** — 认为编码前能充分理解问题空间。对探索性/创新性工作不成立，部分需求只有进入实现阶段才能被真正理解。
2. **上下文无损假设** — 阶段间转换（brainstorm → plan → implement → verify）依赖手写文档传递关键信息。实践中每次转换都是 lossy 的，尤其是"为什么这么做"的隐性决策知识最易丢失，导致后续阶段做出与原始意图矛盾的局部决策。
3. **理性执行者假设** — 假设开发者会忠实地按流程执行。实际中人在认知负荷高时会 skip steps 或 short-circuit，方法论越重，被规避的概率越高。

在我的项目中，**上下文无损假设**是最大的痛点——跨阶段的信息衰减是 subagent 跑偏的首要原因。解决方案是让每个阶段产出物同时作为下一阶段的输入和验证依据，而非仅作为"文档存档"。

- [x] 你对当前 AI4SE 工具与方法论的整体看法。

AI4SE 正处于从"玩具"到"工具"的拐点

**方法论范式的转变：**
软件工程长期处于尴尬位置——大量经典方法论（如形式化规约、设计评审）被实践界视为"形式主义"，因其流程成本高于带来的收益。AI 的加入逆转了这一成本结构：
- 原本昂贵的流程活动（写 spec、生成测试、代码审查）被 AI 自动化，成本降低 1-2 个数量级。
- 流程规划和执行的可编程性，使得方法论的精细化实施首次成为现实。

**对开发者角色的影响：**
Agent 工具正在将软件工程方法论深度融入开发过程，但带来一个悖论：工具链的抽象层次在提升（从"实现什么"到"规约什么"），而工程师的认知视角却被推向更高层次的抽象——从 **"show me the code"** 转向 **"show me the spec"**。这要求工程师具备更强的需求分析、系统设计和验证能力，而非仅仅是编码能力。

**可能的未来：**
- 工作量度量标准从代码行数转向完成的需求数量。
- 流程规划能力（而非编码速度）成为工程师的核心竞争力。
- AI4SE 的真正瓶颈不再是代码生成质量，而是需求获取与规约形式化的能力。
