# 从零搭建南大电费监控系统：一段 AI4SE 工具链的实践记录

> **摘要**: 本文记录了一个真实项目从需求探索到功能完善的完整开发历程，以及在各个阶段中与 AI 编程工具（Claude、OpenCode/Speckit、Superpowers）协作的经验与反思。项目覆盖数据采集、前端可视化、架构重构、自动化部署等多个环节，历时约三周，经历了从纯手工探索到结构化 AI 协作的演进。

---

## 一、背景

**项目简介**:
- 南京大学宿舍电费查询与监控系统
- 数据源：epay.nju.edu.cn 缴费系统
- 部署方式：GitHub Pages（纯静态）
- 技术栈：Python（数据采集）+ Vanilla JS（前端）

**本文的价值**: 与大多数 AI4SE 文章不同，本文不介绍某个特定工具的用法，而是记录**在一个真实项目中，AI 工具如何在不同的开发阶段扮演不同角色**——从 API 逆向、数据建模、前端架构、自动化部署到代码审查，每个阶段我们都面临了不同的挑战，也暴露了 AI 的不同的能力上限与盲区。

## 二、开发历程

### 阶段一：项目探索期（May 14-15）

**关键词**: API 逆向、Playwright MCP、JS 逆向、Cookie 认证、OpenCode + Speckit

**故事**:

项目最初在起源于 epay.nju.edu.cn 缴费系统一无所知——它使用什么 API？如何认证？返回什么格式的数据？

Agent 通过 Playwright MCP 操作浏览器，手动登录 epay.nju.edu.cn，利用浏览器开发者工具捕获网络请求，逐步逆向出电费查询 API 的请求格式和参数。在这个过程中，我们识别出 `auth.nju.edu.cn` 的统一认证登录流程——用户先通过南大统一身份认证登录，再重定向到电费系统——以及 `JSESSIONID`、`MOD_AUTH_CAS`、`_ga` 等关键 Cookie 字段。

从 HTML 中的 `this.check = {...}` JS 片段中，Agent 用正则匹配提取楼栋、房间、余额等字段，验证了解析方案的可行性。随后 Agent 将同步脚本改造为 async 批量查询（`aiohttp` + `asyncio.as_completed`），支持通过命令行批量查询多个房间。

最终产出 `nju_electric_query.py` 初版：包含了 Cookie 认证框架、并发查询、HTML 解析和文件输出的完整数据管道。

**💡 关于 AI 的思考：Agent 在 API 逆向中的角色**

早期项目探索阶段中，为了探索想法的可行性，我们专门编写了一个完整的查询全流程skills，从登陆 `authserver.nju.edu.cn` 获取统一身份认证票据，再到携带票据重定向至 `epay.nju.edu.cn` 电费查询页面，最后从 HTML 中提取余额信息。

Agent 借助 [js-reverse-automation--skill](https://github.com/Fausto-404/js-reverse-automation--skill)（一个前端 JS 逆向自动化技能包，提供 Chrome DevTools MCP 连接、Runtime Hook 探测、加密算法识别与 JSRPC 注入等能力），完成了 `authserver.nju.edu.cn` 登录全流程的逆向：

1. **定位登录入口**: 识别出登录页面的表单参数（`lt`、`execution`、`_eventId`、`rmShown`）及其获取方式
2. **分析密码加密**: 通过 Hook 发现 `pwdEncryptSalt` 的取值和 **AES-CBC 加密**过程——密码先拼接 64 位随机前缀，再用 salt 作为 key 进行 AES 加密，最后 Base64 编码
3. **验证码接口识别**: 找到 `getCaptcha.htl` 接口，接入云码 API 实现自动识别
4. **还原完整登录流程**: 将上述步骤组合为完整的 `nju_auto_login.py`——访问登录页 → 获取参数与验证码 → AES 加密密码 → 提交表单 → Cookie 持久化 → 跳转到电费系统

### 阶段二：数据模型与自动化管道搭建（May 16）— 从脚本到系统

**关键词**: 数据建模、GitHub Actions、Cookie 持久化、OpenCode + Speckit

**故事**:

有了可运行的查询脚本后，下一步是搭建数据模型和自动化管道，让数据采集变成可持续运行的系统。

**数据模型设计**:

Agent 设计了两层数据存储结构：

原始数据层 `database/` 按 `{campus}/{building}/{room_name}/{YYYYMMDD}.json` 组织，每个 JSON 文件包含单次查询的结果，字段由 `nju_electric_query.py` 直接输出：`balance`（余额）、`room_name`（房间名）、`building`（楼栋）、`campus`（校区）、`timestamp`（查询时间）等。这种按时间 + 地理位置分层的设计确保了数据文件在文件系统中自然隔离，便于按校区/楼栋/日期维度检索。

聚合摘要层 `database/summaries/` 按相同的层级组织，但在每个层级上汇聚了统计信息——`overview.json` 包含全局统计（总房间数、总楼栋数、各校区概览），校区摘要包含该校区下各楼栋的汇总数据，楼栋摘要包含该楼栋下各房间的历史 `balance_history`。这样前端在加载时无需遍历全部原始文件，直接从摘要层读取即可。

**数据获取流程**:

数据采集由 GitHub Actions 定时触发（`daily-update.yml`，每日 UTC 0:00），整体流程为：

```
GitHub Actions (scheduled)
    ↓ 加载 Cookie 凭证（secrets.COOKIE_JSON）
    ↓
nju_electric_query.py (aiohttp 并发查询)
    ↓ 输出原始 JSON
database/{campus}/{building}/{room_name}/{date}.json
    ↓
aggregate_data.py (聚合脚本)
    ↓ 输出分层摘要
database/summaries/overview.json / campus / building / room
    ↓
git commit + push → GitHub Pages 更新
```

Cookie 凭证通过 GitHub Secrets 管理，避免了凭证硬编码。`nju_electric_query.py` 使用 `aiohttp` 实现异步并发查询，配合 `asyncio.Semaphore` 控制并发数，确保不会对 epay 服务端造成过大压力。查询完成后自动触发聚合脚本生成前端所需的摘要数据，最后 commit + push 到 GitHub Pages 仓库，前端即可读取最新数据。

这一数据模型的层级结构和获取链路从设计定型后贯穿了项目始终，即使后续经历了 room_id → room_name 的大规模重构，数据模型的基本形态也没有改变。

### 阶段三：前端需求探索阶段（May 17）— 从零开始的可视化

**关键词**: 前端架构、需求梳理、SPA、ECharts、OpenCode + Speckit、122 tasks

**故事**:

前端页面从何而来？整个系统的核心围绕两个概念展开——**电费余额 💰**（还剩多少钱/度）和 **耗电量 ⚡**（用了多少度）。围绕这两个核心指标，系统需要在三个视角上提供数据分析：

- **房间 🏠**: "我"的宿舍还有多少余额？消耗趋势如何？需要充值吗？
- **楼栋 🏢**: 这栋楼的整体用电分布如何？哪些房间耗电异常？我在楼里排第几？
- **校区 🏫**: 不同校区的用电总量对比？各楼栋的消耗排名？

这三个视角构成了前端需求的骨架。

Agent 通过 OpenCode 的 Speckit 框架（`/speckit.specify` → `/speckit.plan` → `/speckit.implement`），从这三个视角出发系统性地产出了完整的用户故事和功能需求。Speckit 将需求拆解为 **122 个任务**，分布在 13 个 Phase 中，覆盖了从数据加载、视图渲染、交互交互到图表可视化的完整链路。

122 个任务最终汇集为 5 个核心视图页面：首页仪表盘（校区概览）、预警中心（低余额/即将耗尽提醒）、楼栋排行榜（高消耗/节能/低余额/突发增长）、楼层热力图（2D 色块展示楼层分布）、多房间对比（折线图叠加比较）。前端的初始架构采用 **SPA 路由**（`#/dashboard`、`#/warnings` 等 hash 路由）和 **ECharts 5.x** 作为可视化引擎。

**好处是**: Speckit 的任务分解机制确保每个用户故事都有对应的实现和验收标准，需求覆盖度很高，122 个任务一步到位，没有遗漏。

**代价是**: 一次性交付大量代码带来了不少问题——ECharts 文件截断（下载的 `echarts.min.js` 仅 44 行，完整 ~100K 行，页面白屏）、null byte 写入污染（`app.js` 第 578 行出现 `\0` 字节，连续修复 4+ 文件）、排行榜排序方向反了（"节能模范"榜首显示最耗电房间）。**更隐蔽的是性能问题**——预警中心串行加载全校 4 个校区、~90 栋楼的 building summary JSON，页面在"加载预警数据..."状态停留超过 10 秒；对比分析页面直接显示"功能开发中"。这些问题虽然最终都被修复，但修复成本随着任务规模的增大而递增。

**💡 关于 AI 的思考：Spec 质量决定了实现的天花板**

Speckit 的任务分解效率很高，但 spec 的清晰度直接决定了产出的质量。122 个任务虽然覆盖全面，但几个本质问题都源于 spec 不够精确：

**"节能模范"排行榜排序方向反了**: Spec 中写了"Top10 节能"这个需求，但没有明确说明排序方向——节能模范应该是"耗电从低到高"（最省电的房间排第一），但 Agent 按照对"排行榜"的默认理解实现了"耗电从高到低"。这不是 Agent 的问题，而是 spec 没有把验收标准写清楚：如果 spec 中明确写了"按耗电量升序排列"或"节能程度 highest → lowest"，就不会出现这个偏差。

**PLAN 断言与实际代码不一致**: 冷启动验证（SPEC_PROCESS §6.2）发现 PLAN 中用"记忆中的 API 行为"编写的断言与实际代码存在偏差——`assert result["room_name"] == "1613"` 但实际 `process_room` 返回的是目录名 `"19栋第16层1613"`；PLAN 要求验证 `overview.json` 的顶层 `total_buildings` 字段，但实际该字段嵌套在 `campuses[name]["buildings_count"]` 中；`merge_room_data` 是公共函数但 PLAN 完全未覆盖其测试。这些错误不是 agent 执行的问题，而是 PLAN 的编写者（人类或 AI）在写 spec/plan 时没有与实际代码交叉验证。Agent 会信任 spec/plan 中的每一行文字——包括错误的示例。

**空状态与错误状态的区分缺失**: Spec 定义了功能需求（"显示房间耗电量"），但没有定义边界状态的分类标准——"这个房间今天还未采集到数据"和"API 返回了错误"在 UI 上应如何区分？Agent 将二者合并处理为"暂无数据"，直到用户验收时才发现两种状态需要不同的视觉提示和交互逻辑。如果在 spec 中补充边界状态的分类和显示规则，就不需要实现后再返工。

**性能需求完全留白**: Spec 定义了"做什么"（如"全校预警扫描"），但没有定义"多快"和"覆盖多少"。Agent 采用最直接的三重嵌套循环——遍历校区 → 遍历楼栋 → 遍历房间，串行 fetch 每个 building summary JSON。实际部署后预警中心需要 ~90 次 HTTP 请求、耗时 10+ 秒才能完成"采样扫描"（每栋楼仅检查前 20 间房）。如果 spec 中明确了加载性能指标（如"预警中心应在 3 秒内完成展示"）或数据覆盖要求（如"扫描全校所有房间"），Agent 会设计不同的架构——例如并行加载楼栋、使用 overview 做快速预筛选，而不是用暴力遍历。

**时间成本是最直接的证据**: Speckit 用 几个小时生成了 122 个任务的代码，但随后花费了 **超过 2 天**（May 18-19）来修复 spec 不清晰导致的问题。Spec 阶段付出 1 小时精炼验收标准，可以节省后续 8+ 小时的 debug 和重构时间。

核心教训是：**在 AI 协作中，Spec 的精度决定了实现的下限。** 模糊的 spec 不会阻止 Agent 完成任务——Agent 会用隐含假设填补空白——但结果往往偏离预期。冷启动验证最有力的证明是：PLAN 中每一行未经交叉验证的代码示例都是一颗定时炸弹。花更多时间精炼 spec 的验收标准、交叉验证关键断言，比花更多时间 debug 要划算得多。

**一个更实用的经验**: 如果你不确定 spec 的质量，就不要用 Speckit。Speckit 的价值在于将清晰的 spec 高效地拆解为可执行的任务，但它也会将模糊的 spec 同样高效地扩散成 122 个方向错误的实现——错误的排序方向、采样的预警扫描、错误的楼层提取，每一个都是 spec 的微小裂缝被 Speckit 放大后的结果。当 spec 还存在疑问时，手动逐个实现、逐个验收，反而能更早发现问题、更灵活地调整方向。

### 阶段四：功能深化与数据完善期（May 19）— Open Design 前端增强

**关键词**: Open Design UI 增强、CSS 变量体系、页面重构

**故事**:

阶段三的问题基本修复后，前端功能已能正常运行，但视觉风格仍停留在"能用"的层面——页面布局和 UI 细节由 Speckit 生成的代码直接决定，缺乏统一的设计语言。用户决定引入 **Open Design** 来提升前端的设计品质。

- **引入 Open Design 设计系统**：从零构建 CSS 变量体系（`--primary`、`--surface`、`--radius` 等），替代 Speckit 生成的内联样式和全局 CSS
- **重构数据服务层**: `js/modules/` 下的 20+ 个模块文件合并为 `js/data-service.js`，统一数据获取和缓存
- **拆分页面**: 独立的 `building-view.html`、`campus-view.html`、`room-view.html`、`electricity-monitor-index.html`，替代单页 SPA 结构
- 净变化：+6041 / -13574 行

这次重构不是功能修复，而是设计上的主动增强，为后续的功能迭代建立了更清晰的代码结构和统一的视觉语言。

**💡 关于 AI 的思考：Open Design 擅长设计，不擅长业务**

Open Design 在处理视觉层面的需求时效果很好——CSS 变量体系让全局风格统一、`data-service.js` 收敛了数据访问层、页面拆分让路由清晰。但遇到业务需求时就力不从心了。最典型的例子是**耗电量直观类比**功能：把抽象的 kWh 转化为"小米 SU7 跑 X 公里""够 ChatGPT 回答 Y 个问题"这类有感知的场景——这需要作者对科技产品有了解、对受众的认知语境有判断、对类比精度有取舍。Open Design 的 `DESIGN.md` 可以定义颜色和间距，但定义不了"什么样的类比能让大学生觉得有趣且准确"。

同样，这个阶段的业务功能——排行榜请求池逻辑（控制并发防阻塞）、校区真实消耗数据计算、IndexedDB 缓存策略——都是 Open Design 无法触及的领域。它们不涉及"好不好看"，只涉及"对不对"。Open Design 的定位是设计工具，不是业务逻辑引擎，理解这个边界才能正确地使用它。

### 阶段五：新需求密集开发期（May 21-30）— 业务功能与 Visual Companion

**关键词**: 耗电量类比、Visual Companion、分布图表、日期回退、并行加载

**故事**:

Open Design 前端重构后，进入了长达 10 天的业务功能密集开发期，完成了两大类需求：

**第一类：业务增强功能（May 21-27）**

Open Design 确立了代码结构和视觉语言，但业务功能仍需逐个实现：

- **耗电量直观类比**: 将抽象的 kWh 转化为有感知的场景——"够 iPhone 16 充满 X 次""≈ 小米 SU7 跑 X 公里""够训练 DeepSeek-R1 回答 X 道高中数学题"。纯业务逻辑，与设计系统无关。
- **全校区排行榜百分比**: 显示房间在全校区的排名百分位，让用户知道自己的用电水平处于什么位置。数据来自跨页面缓存共享。
- **排行榜请求池**: 批量请求时的并发控制，防止短时间内大量 HTTP 请求触发浏览器限制或服务端限流。
- **IndexedDB 缓存迁移**: localStorage → IndexedDB，支持更大的缓存容量和结构化数据存储。迁移过程暴露了 Agent 对同步/异步 API 差异的盲区——localStorage 是同步，IndexedDB 是异步，多处遗漏 `await` 导致返回 Promise 而非数据。
- **智慧排序**: 选择框根据校区、楼栋、房间的常用程度智能排序，优先显示最可能被选中的选项。

**第二类：Visual Companion 需求（May 28-30）**

业务功能基本完成后，进入需要可视化交互的设计阶段，Agent 首次使用 **Visual Companion**——在浏览器中展示实时 mockup 供用户选择：

- **分布图表系统**: Agent 在浏览器中展示 3 种布局方案的实时 mockup（直方图+拟合曲线 / ECDF / 组合），用户直观选择而非想象描述。选中后实现对 4 种分布类型（正态/对数正态/Gamma/双峰 EM）的自动拟合，用 BIC 选择最优模型，设计自适应标注系统（±σ 区间 / IQR / 均值众数 / 双峰峰位）。
- **日期自动回退**: Agent 初提"严格日期匹配，无数据就显示 --"，被用户推翻。随即补提"自动回退 + banner 通知"方案——今日覆盖度不足时自动搜索最近 7 天第一个可用日期，用可关闭的 banner 保持数据诚实。展示了 Agent 被推翻后的设计迭代能力。
- **楼栋并行加载**: 串行加载 O(n) → 并行加载 O(1)，将预警中心 10+ 秒加载时间压缩到秒级。
- **房间 ID 扫描**: 批量发现有效房间 ID 的完整机制——CLI 参数支持扫描区间、指数退避重试、ROOM_NOT_FOUND 不重试、信号安全终止保存。

**💡 关于 AI 的思考：Brainstorming 确定需求 vs Speckit 生成 spec**

Speckit 一次性写 clear spec → 生成代码，但模糊 spec 只会扩散为方向错误的 122 个任务。Brainstorming 不写 spec，通过追问链逐步确定需求。以下是 6 个追问前后的 spec 对比：

**1. 数据完整度定义**

| | 内容 |
|--|------|
| **before** | — 无明确 spec。Claude 追问暴露歧义："**数据完整度怎么定义？是按该栋楼有数据房间数/总房间数，还是按该校区有数据楼栋数/总楼栋数？**" |
| **after** | → 按房间维度：有数据房间数 / 总房间数 |

**2. 日期回退策略**

| | 内容 |
|--|------|
| **before** | — Claude 初提："**我建议在 data-service 中增加严格日期匹配，移除所有 fallback，这样数据更诚实。只有当特定日期没有数据时才显示`--`。**" |
| **after** | → 自动搜索最近 7 天第一个覆盖度 > 50% 的日期，banner 提示 |

**3. 映射文件维护职责**

### 1. 哪些 Superpowers 技能发挥了最大作用？哪些感觉是"形式大于实质"？

**最大作用**:
1. **brainstorming** — 在需求不明确时，能将模糊的初始想法系统地转化为成熟设计。视觉原型的输出帮助提前发现需求矛盾，降低了后期返工成本。
2. **systematic-debugging** — 结构化排查流程显著缩短了根因分析时间，避免了经验主义的盲目试错。
3. **TDD + code review** — 配合 requesting-code-review 构成了有效的质量闭环，在开发阶段拦截了大部分回归缺陷。

**形式大于实质**:
1. **writing-plans** — plan 阶段往往已包含具体代码片段，导致后续实现退化为机械的"替换粘贴"任务。plan 的价值被上游 spec 和 brainstorming 阶段的细节程度挤压，边际收益递减。

### 2. TDD 在 AI 协作下的体感如何？

在单次 Superpowers 流程内，TDD 有效消除了显性缺陷，测试先行迫使 AI 明确行为边界。但在跨流程（多轮迭代或多 agent 协作）场景中，TDD 反而引入摩擦：前一轮生成的测试可能在后一轮架构调整后失效，维护测试与生产代码的一致性成为额外开销。

结论：**TDD 是局部放大器、全局摩擦源**，需要更精细的测试范围管理策略。

### 3. SPEC 与 PLAN 的质量如何影响了后续实现？

1. **Spec 不清晰 → 实现偏差与返工**。模糊的规约迫使 subagent 依赖隐含假设填补空白，最终输出与预期产生系统性偏离。典型案例：前端交互行为的 spec 未明确"空状态"和"错误状态"的区分，subagent 将二者合并处理，导致后续验收阶段才发现设计意图未被满足。

2. **Plan 不完整 → 架构缺陷与技术债务**。缺少边界条件、错误路径或性能约束的 plan，在实现阶段暴露问题，修复成本呈指数级增长（验证了 Boehm 曲线）。具体表现：subagent 会选择"最短路"实现方案，而非"最鲁棒"方案。

核心结论：**Spec 决定"做对的事"，Plan 决定"把事做对"。** 两者中的任何缺位都会使 subagent 的执行质量退化到纯模型先验的水平。

### 4. 最有效的 prompt / context 策略是什么？

前端开发场景中，先向 AI 精确描述正确的交互逻辑（而非直接要求生成代码），再让其通过 MCP 操作浏览器进行端到端验证。这一策略有效的根本原因在于：**它将 AI 的注意力从"怎么写代码"引导到"怎么验证行为正确性"**，通过自验证循环确保生成代码与需求语义一致，减少了对 prompt 措辞敏感性的依赖。

### 5. 如果重做，会改变哪些做法？

1. **在 122 tasks 交付后立即引入 code review** — 分布式扫描竞态条件在 brainstorming、writing-plans、implementation 三阶段都未被发现，仅 code review 暴露。如果 Speckit 阶段之后立即做 code review，可以更早建立质量意识。

2. **在更早的阶段引入 brainstorming** — 阶段三的 Speckit 122 tasks 交付了大量代码，但排序方向、采样扫描、楼层解析等 bug 都源于 spec 模糊。如果先通过 brainstorming 澄清这些模糊点再让 Speckit 生成代码，可以避免大量返工。

3. **阶段一就建立 room_name 映射体系** — 项目早期用 room_id 做主键，后续不得不花一个完整阶段重构为 room_name + 映射文件。如果在 curl 探索 API 的阶段就设计了映射体系，可以减少一次全量数据迁移。

## 四、总结：对 AI4SE 工具链的反思

### 4.1 Superpowers 方法论的前提假设

Superpowers 方法论包含若干未明言的假设，这些假设在实践中部分成立、部分存疑：

1. **可预知性假设** — 认为编码前能充分理解问题空间。对探索性/创新性工作不成立，部分需求只有进入实现阶段才能被真正理解。
2. **上下文无损假设** — 阶段间转换（brainstorm → plan → implement → verify）依赖手写文档传递关键信息。实践中每次转换都是 lossy 的，尤其是"为什么这么做"的隐性决策知识最易丢失，导致后续阶段做出与原始意图矛盾的局部决策。
3. **理性执行者假设** — 假设开发者会忠实地按流程执行。实际中人在认知负荷高时会 skip steps 或 short-circuit，方法论越重，被规避的概率越高。

在我的项目中，**上下文无损假设**是最大的痛点——跨阶段的信息衰减是 subagent 跑偏的首要原因。解决方案是让每个阶段产出物同时作为下一阶段的输入和验证依据，而非仅作为"文档存档"。

### 4.2 对未来 AI4SE 的展望

AI4SE 正处于从"玩具"到"工具"的拐点：

**方法论范式的转变：** 软件工程长期处于尴尬位置——大量经典方法论（如形式化规约、设计评审）被实践界视为"形式主义"，因其流程成本高于带来的收益。AI 的加入逆转了这一成本结构——原本昂贵的流程活动（写 spec、生成测试、代码审查）被 AI 自动化，成本降低 1-2 个数量级。

**对开发者角色的影响：** Agent 工具正在将软件工程方法论深度融入开发过程，但带来一个悖论：工具链的抽象层次在提升（从"实现什么"到"规约什么"），而工程师的认知视角却被推向更高层次的抽象——从 **"show me the code"** 转向 **"show me the spec"**。这要求工程师具备更强的需求分析、系统设计和验证能力，而非仅仅是编码能力。

**可能的未来：**
- 工作量度量标准从代码行数转向完成的需求数量。
- 流程规划能力（而非编码速度）成为工程师的核心竞争力。
- AI4SE 的真正瓶颈不再是代码生成质量，而是需求获取与规约形式化的能力。
