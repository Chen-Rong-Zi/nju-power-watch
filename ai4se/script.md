# 阶段一：项目探索期 — 讲稿

## 南京大学的统一认证体系

南京大学几乎所有的门户网站都挂在 **authserver.nju.edu.cn** 这个统一身份认证平台上。无论是教务系统、财务系统、还是我们今天要攻克的电费查询系统，全部走同一套认证流程。

你在页面上看到的这张截图，就是 authserver 的登录页面。它长得很普通——用户名、密码、验证码——但背后的加密机制并不简单。

## 电费查询系统的认证壁垒

我们的目标是 **epay.nju.edu.cn** 这个电费查询网站。但它不对外开放，你必须先通过 authserver 的统一认证，拿到 Cookie，然后带着这个凭证才能访问电费数据。

这就带来了第一个问题：**我们不知道认证流程的具体细节**。

- 密码是怎么加密的？（明文传输不可能）
- 验证码是怎么生成的？
- 登录成功后的 Cookie 长什么样？
- 拿到 Cookie 后怎么调用电费查询 API？

这些信息全部隐藏在浏览器的网络请求和 JavaScript 代码中，肉眼看不到，必须动手逆向。

## 逆向过程

为了解决这些问题，Agent 借助了 **js-reverse-automation skill**——这是一个专门做前端 JS 逆向的自动化工具集，GitHub 上 446 Stars，v2.0 版本。

它的工作流程分 9 个 Phase（P0-P8）：

1. **P0-P1**: 设置 Chrome DevTools MCP 连接，建立 Hook 探针
2. **P2-P3**: 分析加密算法，识别 Webpack 模块
3. **P4-P5**: 7 维度评分验证候选函数，注入 JSRPC
4. **P6-P7**: 启动 Flask 代理，对接 Burp Suite autoDecoder
5. **P8**: 交付可运行的工程代码

映射到我们的 NJU 实战中：

**第一步，定位登录入口。** Agent 通过浏览器开发者工具捕获表单提交请求，识别出 `lt`、`execution`、`_eventId`、`rmShown` 这几个隐藏参数——它们是 CAS 协议的一部分，用于防止 CSRF 攻击和跟踪会话状态。

**第二步，Hook 密码加密。** 这是最关键的一步。Agent 在运行时注入 Hook 探针，捕获到了 `pwdEncryptSalt` 这个变量，进而发现了完整的加密链路：密码先拼接 64 位随机前缀，然后用 salt 作为 key 进行 **AES-CBC 加密**，最后 Base64 编码。如果你看一下页面上的代码片段，这就是核心逻辑——不到 10 行 Python，但从发现到验证花了一整个阶段。

**第三步，验证码识别。** authserver 的验证码通过 `getCaptcha.htl` 接口获取。Agent 接入了云码 API 实现自动识别，使得整个登录流程可以无人值守地完成。

**第四步，还原完整流程。** 将上述步骤组合起来，就是 `nju_auto_login.py`——从访问登录页、获取参数与验证码、AES 加密密码、提交表单，到 Cookie 持久化、跳转到电费系统，一气呵成。

## 最终产出

有了登录凭证之后，Agent 将同步的查询脚本改造成了异步批量版本：

- **aiohttp** 实现并发 HTTP 请求
- **asyncio.Semaphore** 控制并发数（默认 10）
- **指数退避重试** 应对服务端限流

最终产出了 `nju_electric_query.py` 的第一个可用版本。

## 关于 AI 的思考

这个阶段展示了 AI 一个非常突出的能力：**在人类指导方向的前提下，Agent 可以独立执行细节层面的逆向工程**。传统做法是：开发者手动打开 Chrome DevTools，逐个请求查看、逐行 JS 代码分析、手动编写解密脚本。而 Agent 借助 js-reverse-automation skill，在几个小时内完成了从"不知道有什么 API"到"可运行的查询脚本"的全链路交付。

但有一个前提不可忽视：**需要人类来判断"下一步该做什么"**。Agent 擅长执行，不擅长决策方向。
