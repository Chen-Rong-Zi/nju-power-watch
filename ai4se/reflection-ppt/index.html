def encrypt_password(pwd, salt):
    raw = random_hex(64) + pwd
    cipher = AES.new(salt.encode(),
      AES.MODE_CBC, iv=salt[:16].encode())
    return b64encode(
      cipher.encrypt(pad(raw))).decode()

阶段一讲稿：项目探索期

南京大学的统一认证体系

南京大学几乎所有的门户网站都挂在 authserver.nju.edu.cn 这个统一身份认证平台上。无论是教务系统、财务系统、还是我们今天要攻克的电费查询系统，全部走同一套认证流程。

你在页面上看到的这张截图，就是 authserver 的登录页面。它长得很普通——用户名、密码、验证码——但背后的加密机制并不简单。

电费查询系统的认证壁垒

我们的目标是 epay.nju.edu.cn 这个电费查询网站。但它不对外开放，你必须先通过 authserver 的统一认证，拿到 Cookie，然后带着这个凭证才能访问电费数据。

这就带来了第一个问题：我们不知道认证流程的具体细节。密码是怎么加密的？验证码是怎么生成的？登录成功后的 Cookie 长什么样？这些信息全部隐藏在浏览器的网络请求和 JavaScript 代码中，必须动手逆向。

逆向过程

为了解决这些问题，Agent 借助了 js-reverse-automation skill——这是一个专门做前端 JS 逆向的自动化工具集，GitHub 上 446 Stars，v2.0 版本。它的工作流程分 9 个 Phase（P0-P8），从设置 MCP 连接、注入 Hook 探针，到分析加密算法、评分验证，再到 JSRPC 注入和 Burp 对接，最终交付可运行的工程代码。

映射到 NJU 实战中：第一步，定位登录入口——识别 lt、execution、_eventId 等隐藏参数。第二步，Hook 密码加密——捕获 pwdEncryptSalt，发现 AES-CBC 加密链路，密码先拼接 64 位随机前缀，再用 salt 加密后 Base64 编码。第三步，验证码识别——接入云码 API 实现自动识别。第四步，还原完整流程——从登录页到 Cookie 持久化再到电费系统，一气呵成。

最终产出

有了登录凭证之后，Agent 将同步脚本改造成了异步批量版本——aiohttp 并发、Semaphore 限流、指数退避重试，产出了 nju_electric_query.py 的第一个可用版本。

关于 AI 的思考

这个阶段展示了 AI 一个非常突出的能力：在人类指导方向的前提下，Agent 可以独立执行细节层面的逆向工程。传统做法需要开发者手动分析每个请求和每行 JS 代码，而 Agent 借助 skill 在几个小时内完成了从"不知道有什么 API"到"可运行脚本"的全链路交付。但有一个前提不可忽视：需要人类来判断"下一步该做什么"。Agent 擅长执行，不擅长决策方向。

阶段二讲稿：数据模型与自动化管道

从脚本到系统

有了可运行的查询脚本之后，下一步不是急着写前端，而是先把数据模型和自动化管道搭好。让数据采集从"手动跑脚本"变成"可持续运行的系统"。

数据管道

整个数据采集由 GitHub Actions 定时触发——每天 UTC 2:00 自动运行。流程是这样的：首先 nju_auto_login.py 用凭据登录 authserver.nju.edu.cn，经过 AES-CBC 加密认证拿到 Cookie 文件。然后 nju_electric_query.py 用这个 Cookie 并发查询所有房间，控制并发数防止触发服务端限流。查询完成后，aggregate_data.py 聚合生成分层摘要，最后自动 commit 并 push 到 GitHub Pages，前端就能读到最新数据了。

两层数据模型

这里最核心的设计是两层存储结构。原始数据层按校区→楼栋→房间→日期的路径组织，每个 JSON 文件就是一次查询的结果，append-only 写入。聚合摘要层按相同的层级组织，但在每一层上都预先算好了统计信息——overview.json 有全局统计，校区摘要含各楼栋汇总，楼栋摘要含各房间的 balance_history。这样前端加载时不需要遍历几千个原始文件，直接从摘要层读就行。

为什么这个设计能贯穿始终

这套模型从 5 月 16 日定型后，经历了 room_id 到 room_name 的大规模重构、前端三次重写、新增十几个功能模块——但数据模型的基本形态一次都没改过。原因在于：它把"数据怎么存"和"数据怎么用"清晰地分开了。原始层只管记录，摘要层只管查询，各自独立演化。

这个阶段的教训是：在 AI 协作中，先让 Agent 定义数据契约再实现功能，比先写代码再补数据结构要高效得多。Agent 在数据建模上展现的架构直觉，其实比在写具体功能代码时更可靠。

💡 关于 Speckit 的思考

Speckit 用 Story 描述"用户要什么"，用 Spec 描述"系统做什么"。Speckit 尝试将用户需求细化为专业的 Spec，但由于缺乏 Brainstorming 追问环节，没有真实的用户反馈来校准 Spec 的完整性，往往效果不尽人意：Agent 忠实执行了 Spec，但 Spec 本身可能已经偏离了用户的真实意图。

阶段三讲稿：前端需求探索阶段

三个视角的设计动机

整个前端系统的核心围绕两个概念展开——电费余额还剩多少钱和耗电量用了多少度。围绕这两个核心指标，系统需要在三个视角上提供数据分析。

房间视角是最直接的——"我"的宿舍还有多少余额？消耗趋势如何？需要充值吗？这是每个学生最关心的问题。楼栋视角则把视野拉高到整栋楼——这栋楼的整体用电分布如何？哪些房间耗电异常？其他同学的平均用电水平是多少？我在楼里排第几？校区视角是对全校的整体把控——不同校区的用电总量对比如何？各楼栋的消耗排名怎样？有没有需要预警的低余额楼栋？

这三个视角构成了前端需求的骨架。Agent 通过 Speckit 框架，从这三个视角出发，系统性地产出了完整的用户故事和功能需求。

Speckit 框架的高效与代价

Speckit 将需求拆解为 122 个任务，分布在 13 个 Phase 中，覆盖从数据加载、视图渲染到交互和图表可视化的完整链路。最终汇集为 5 个核心视图页面——首页仪表盘、预警中心、楼栋排行榜、楼层热力图、多房间对比。

好处是覆盖率很高，122 个任务一步到位，没有遗漏。但代价是一次性交付大量代码带来了不少问题——ECharts 文件截断导致白屏、排行榜排序方向反了、预警中心串行加载 10 秒以上。

Spec 不清晰的四个典型案例

看这几个例子。"Top10 节能"——Spec 只写了这四个字，Agent 按照对"排行榜"的默认理解实现了耗电从高到低，结果节能模范的榜首显示的是最费电的房间。如果 spec 明确写了"按耗电量升序排列"，就不会有这个偏差。

"显示房间耗电量"——Spec 没有定义边界状态。"这个房间今天还没采集到数据"和"API 返回了错误"在 UI 上应该怎么区分？Agent 合并处理为"暂无数据"，直到验收才发现两种状态需要不同的视觉提示。

"全校预警扫描"——Spec 定义了做什么，但没有定义多快。Agent 用三重嵌套循环串行请求，预警中心需要 90 多次 HTTP 请求、耗时 10 秒以上。如果 spec 明确了"3 秒内完成"，Agent 会设计完全不同的架构。

最隐蔽的是——有的需求甚至没有 spec。数据完整度的标准完全没有定义，Agent 只能用自己的假设填补。

Brainstorming 的价值

这些问题的共性是：问题在写 spec 时不可见，只有在尝试回答时才会浮现。Brainstorming 通过追问链让不可见的问题在实现前变得可见——"按升序还是降序？""空状态和错误状态怎么区分？""性能指标是多少？"——这些追问在 Speckit 阶段从未被提出，所以问题直到验收才被发现。

修正成本是最直接的证据：122 个任务的代码生成用了几个小时，但修复 spec 不清晰导致的问题花了超过 2 天。核心教训是——如果你不确定 spec 的质量，就不要用 Speckit。手动逐个实现、逐个验收，反而能更早发现问题。

Open Design 在处理视觉层面需求时效果很好——CSS 变量体系让全局风格统一、data-service.js 收敛数据访问层、页面拆分让路由清晰。但遇到业务需求时就力不从心了：kWh 直观类比需要作者对科技产品有了解、对受众认知语境有判断；IndexedDB 迁移暴露了 Agent 对同步/异步 API 差异的盲区。Open Design 的定位是设计工具，不是业务逻辑引擎——理解这个边界才能正确地使用它。

阶段四讲稿：功能深化与数据完善期

为什么需要 Open Design

阶段三的 Speckit 122 tasks 交付了完整的前端功能，但视觉风格停留在"能用"的层面——页面布局和 UI 细节由 Speckit 生成的代码直接决定，CSS 变量体系不存在，内联样式和全局 CSS 混用，缺乏统一的设计语言。

Open Design 做了什么

第一，从零构建了完整的 CSS 变量体系。用 oklch 色域定义了 --accent、--surface、--radius、--shadow、--font-display 等设计 Token，全局风格从此统一。第二，数据服务层重构——将 20+ 散落模块合并为 data-service.js，统一数据获取和缓存。第三，页面拆分——独立的 building-view、campus-view、room-view、electricity-monitor-index，替代单页 SPA 结构。

净变化：+6041 / -13574 行。这是主动的设计增强，不是被动修复。

Open Design 的边界

但 Open Design 遇到业务需求时就力不从心了。最典型的例子是耗电量直观类比——把抽象的 kWh 转化为"小米 SU7 跑 X 公里""够 ChatGPT 回答 Y 个问题"这类有感知的场景。这需要作者对科技产品有了解、对受众的认知语境有判断、对类比精度有取舍。Open Design 的 DESIGN.md 可以定义颜色和间距，但定义不了"什么样的类比能让大学生觉得有趣且准确"。

同样，排行榜请求池并发控制、IndexedDB 缓存迁移、校区真实消耗数据计算——这些不涉及"好不好看"，只涉及"对不对"。Open Design 的定位是设计工具，不是业务逻辑引擎。理解这个边界才能正确地使用它。

💡 核心洞察

🎯 Spec 质量决定实现的天花板
Speckit 的任务分解效率很高，但 spec 的清晰度直接决定了产出的质量。模糊的规约迫使 Agent 依赖隐含假设填补空白——排序方向、错误状态区分、性能目标，这些在 spec 中没有明确写明的细节最终都成为了 bug。核心教训：如果你不确定 spec 的质量，就不要用 Speckit。
🔄 Brainstorming 让不可见的问题变得可见
122 个任务的代码生成用了几个小时，但修复 spec 不清晰导致的问题花了超过 2 天。Brainstorming 通过追问链让不可见的问题在实现前变得可见——"按升序还是降序？""空状态和错误状态怎么区分？""性能指标是多少？"——这些追问在 Speckit 阶段从未被提出。
👁️ Visual Companion 降低沟通成本
用户选择方案 C 只用了 3 秒——如果用文字描述来回沟通至少需要 3 轮。Visual Companion 将设计选择从"你想象一下"变为"你看一下"，极大降低了沟通成本。
🧩 AI 的盲区需要结构性补偿
运行时语义盲区：Chart.js Proxy 死循环、坐标类型不匹配
边界情况盲区：负余额、最后 bin、同名房间去重
并发环境盲区：workflow 竞态条件、数据一致性

systematic-debugging 和 code review 是当前最有效的补偿手段。

Open Design 擅长设计，不擅长业务——理解这个边界才能正确地使用它。

AI4SE 不是取代开发者，而是改变开发者需要关注的事情。

总结讲稿：AI4SE 工具链的实践反思

项目概况

整个项目历时约三周，经历了三种协作范式：纯 Claude 交互、OpenCode /speckit、Superpowers。净增 +19,032 行，删除 -23,014 行，共 16 轮 Superpowers 流程。

三种协作范式的演进

第一阶段是纯 Claude 交互，无结构化流程。第二阶段引入 OpenCode /speckit，实现了 spec→plan→implement 的流程化。第三阶段使用 Superpowers，增加了 brainstorming→verify 环节，形成了完整的闭环。

核心洞察

第一，Spec 质量决定实现的天花板——模糊的规约迫使 Agent 依赖隐含假设。第二，Brainstorming 让不可见的问题变得可见——追问链能在实现前发现问题。第三，Visual Companion 将设计选择从"你想象一下"变为"你看一下"，极大降低了沟通成本。第四，AI 的盲区需要结构性补偿——systematic-debugging 和 code review 是最有效的手段。
