## Slide 2: 项目介绍

先说下做这个项目的动机。
    五月的南京已经很热了，宿舍空调不得不开，但大家心里都没底——上个月电费还剩多少？这个月够不够用？半夜突然停电才发现余额不足，这种事情在宿舍里太常见了。而且通过南大app查看,步骤繁琐。就算看到了余额数字，也无法知道当日消耗的费用以及余额还能使用多久
    为了解决以上痛点, 这个项目部署了自动爬虫, 获取每日余额数据并提供了房间,楼栋,校区三个视角的数据分析情况, 给同学们对于电费一个直观的感受
    该项目目前托管在github pages上, 并在github上收获了同学们的11个star和多个issue, 同学们可以通过njupower.top访问,如果觉得有用也希望可以点一个star支持一下
    接下来介绍一下这个项目的几个开发阶段

## Slide 3: 阶段一 · 项目探索期

第一阶段是项目探索期——目标是拿到电费系统的数据。
  同学们了解过南大app的大致架构吗? 我第一次安装的时候以为这是一个安卓原生的app，但其实南大 App 本质上就是一个浏览器，或者说是一个门户网站。里面查成绩、查课表、查电费这些按钮，每一个点进去都是打开一个独立的网页, 在电脑中使用浏览器也可以打开。这些网页全部挂在统一认证网关后面。
  意味着只要在统一认证系统中登录一次、拿到 Cookie，就等于拿到了访问所有网页的通行证。电费系统也不例外。
  统一身份认证手动登录一次很简单，但目标是自动化——让脚本每天自动登录、自动查询。这就必须搞清楚 authserver 的认证流程：密码怎么加密的？验证码怎么识别的？
  我找到了一个专门做 js 逆向的skill——js-reverse-automation。它赋予了 Agent 在浏览器中逆向分析加密流程的能力——自动注入 Hook 探针、捕获加密函数、还原加密算法。
  Agent 先分析了 authserver 登录页面的表单提交过程，在浏览器中 Hook 住了密码加密函数，发现是 AES-CBC 加密——密码会先拼接 64 位随机前缀，再用 salt 加密后 Base64 编码。接着接入了第三方 API 实现验证码自动识别，最终成功模拟了完整的登录流程，拿到了有效的 Cookie。

## Slide 4: 阶段二 · 数据模型与自动化管道

拿到 Cookie 之后，查询数据的脚本就很容易完成了。但需求是每天自动抓取数据，而不是手动跑脚本——Agent 告诉我 GitHub Actions 可以创建定时任务，并设计了下面的流水线配置。
    左边是 GitHub Actions 的工作流配置。每天 UTC 2:00 自动触发。
    中间是完整的数据采集流水线：携带上一个step拿到的 Cookie 即可通过验证，然后并发查询全校所有房间，接着按校区→楼栋→房间的层级聚合生成摘要，最后自动 commit 到仓库并部署到到 GitHub Pages。
    GitHub Actions 的定时任务还有很多有趣的用法——比如自动抓取推送每日科技新闻、自动签到等等。感兴趣的同学也可以去试试。

## Slide 5: 阶段三 · 前端需求探索阶段

第三阶段是前端需求探索。
    这个项目设计的前端需求围绕`电费余额`与`耗电量`两个核心概念, 分别在 房间 / 楼栋 / 校区 三个视角提供数据分析。
    这个阶段在项目要求发布前完成, 所以这一阶段我使用的是 opencode + speckit 并未使用 superpowers。
    Agent 通过 Speckit 框架将需求拆解为 10 份 User Stories、122 个任务。但效果不尽人意——大多数规格最终都需要重新来解释。前期需求不明确的阶段，Agent 只能用自己的假设填补空白，比如"Top10 节能"没说排序方向，Agent 默认降序，结果榜首是最费电的房间；"加载和错误处理"没定义边界状态，"未采集"和"API 报错"被合并处理，直到验收才发现问题。
    Speckit 用 Story 描述"用户要什么"，用 Spec 描述"系统做什么", speckit将story和spec分离, 似乎想从用户需求中提炼出具体的软件规格。但由于缺乏 Brainstorming 追问环节，没有真实的用户反馈来校准 Spec 的完整性，Spec 本身可能已经偏离了用户的真实意图。122 个任务的代码生成用了几个小时，但修复 spec 不清晰导致的问题花了超过 2 天。

## Slide 7: 阶段五 · 新需求密集开发期

阶段五讲稿：新需求密集开发期
    工具转变
    第五阶段开始，工具使用转向了 Claude + Superpowers。严格遵循完整流程——Brainstorming 阶段澄清需求，实现阶段走 TDD，完成后还有 Review 验收。
    左侧的几个新功能——耗电量直观类比、排名百分位、请求池并发控制、IndexedDB 缓存迁移——大部分都只需要一轮 Superpowers 就能完成。验收时基本不用返工。
    在这个阶段我发现了superpowers相比speckit更好的一点
    最初的有一个增加当前房间在楼栋消耗分布的需求——"加个分布图，让用户了解当日耗电量位于本楼栋中的什么位置"
    但 claude 在 Brainstorming 阶段根据我的需求 提出可以用不同的统计模型（比如正态分布）来拟合楼栋的耗电量分布，让系统自动用 BIC 选出最匹配的统计模型。
    让用户不仅能看到"我在什么位置"，还能看出整栋楼的耗电模式是正态分布还是偏态分布，有没有异常聚类。claude最终把我的模糊的想法转化成了可落地的规格。

## Slide 9: Deadline Gate · 为什么需要时间门禁

最后是这段开发过程里面根据实际情况构建出的skills
    同学们有没有遇到过这种情况——你给了你的agent所有权限让它跑了一整个晚上，第二天早上发现它什么都没完成，甚至让事情变得更糟了？
    那期待中agent的工作模式是什么?我认为是agent使用"有限资源", 在"一定假设"内可以完成任务, 如果agent使用了超过计划中的资源, 他应该停下来反思, 是否偏离计划, 以及假设是否被破坏.
    这里的资源可以代表时间, token, 上下文余量等可以计算的量, 而"一定假设"则代表了任务计划里面假设存在的情况(比如网络通畅, 认证通过, 外部资源可用等)
    在上面的情况里面也许是agent偏离了计划, 导致了使用过量资源, 也许是某些假设被破坏(比如无法下载依赖的包), 才导致任务无法被实现
    为了杜绝这两种情况, 我根据agent"是否在有限资源能结束任务"和"结束的任务是否被正确完成"划分了四种情况
    左边红色代表"没能在有限资源内结束任务"，右边绿色则代表"能够在有限资源内结束任务"的区域。
    它们的交集则代表了"能够结束任务, 但无法正确完成"
    我希望可以harness来彻底杜绝左边红色减去交集的情况
    至于交集的这部分我认为取决于 Context Engineering，也就是 计划, spec 写得清不清楚、prompt 设计得好不好才能解决。
    所以我设计了一个skills, 强制agent事先计划并给每一个子任务设计定时器, 如果执行任务中定时器结束, 则告诉agent反思是否偏离计划或者某些假设被打破
    有同学可能会问, 上面的分析中假设了agent会先制定计划在执行, 但是所以任务一定需要计划吗?
    右边是一个真实的教训。有一次我让 Agent 追踪一个新文件并 push 到远端，特别强调了其他未追踪的文件不要动。Agent 觉得太简单了，几行命令的事，不需要计划。结果 push 时发现远端有冲突，Agent 自动执行 rebase，过程中误操作把本地其他未追踪文件删除了。等我发现的时候已经找不回来了。
    这个案例就落在"出问题"区域——Agent 超出了资源边界却不自知。所以 Deadline Gate 就是给 Agent 装上的边界检测器，超限就停，等人介入。

## Slide 10: Deadline Gate · 机制与实现

核心流程六步：估算 → 设截止 → 执行 → 到期 → 停止 + 分析 → 汇报 + 人类介入。关键在"到期"和"停止"之间没有缓冲——没有"再给我一分钟"。
    四个 Red Flag 是实际使用中 Agent 最常用的借口："很简单"、"流程比做事慢"、"先动手再补计划"、"设了定时器就算走流程了"——看到就要警惕。
    实现上两个平台不同。Claude Code 通过 CronCreate 设定定时器，到期后提醒 Claude 停止并汇报，是软约束。opencode 更彻底——插件通过 set_timeout 设定时间，到期后 session.abort() 强制打断当前执行，包括正在跑的 bash 命令，再注入 prompt 要求分析汇报，同时工具钩子拦截后续所有调用。
    适用场景是有明确边界、步骤可预期的任务：GitHub 复现、CI 修复、批量脚本、部署流程。探索性研究不适用——API 调研、架构设计这些没有明确的"完成"定义。
