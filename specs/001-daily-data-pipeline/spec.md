## User Scenarios & Testing *(mandatory)*

### User Story 1 - Daily Automated Data Collection (Priority: P1)

As a system administrator, I need the electricity consumption data to be collected automatically every day so that the frontend dashboard always displays up-to-date information without manual intervention.

**Why this priority**: Without automated collection, users must manually run scripts, which is error-prone and leads to stale data. This is the foundation for all other functionality.

**Independent Test**: Can be fully tested by triggering the GitHub Action workflow manually and verifying that new daily JSON files are created in the database directory for all tracked rooms.

**Acceptance Scenarios**:

1. **Given** the GitHub Action workflow is configured, **When** the scheduled time arrives (or manual trigger), **Then** all registered room IDs are queried and results saved to dated JSON files
2. **Given** a successful daily run completes, **When** checking the database directory, **Then** each room has a new JSON file with today's date containing electricity data
3. **Given** the authentication cookie expires, **When** the daily job runs, **Then** the job fails with a clear error message indicating cookie renewal is needed

---

### User Story 2 - Data Retention and Storage Optimization (Priority: P2)

As a system administrator, I need an automated data cleanup strategy that balances storage costs with analytical value so that the repository remains maintainable while preserving sufficient historical data for trend analysis.

**Why this priority**: Without cleanup, daily data accumulation will bloat the repository. This becomes critical after several months of operation.

**Independent Test**: Can be tested by running the cleanup script on a test dataset with various retention periods and verifying that old daily files are archived/compressed while recent data remains accessible.

**Acceptance Scenarios**:

1. **Given** daily data older than the retention period exists, **When** the cleanup job runs, **Then** old daily JSON files are compressed into monthly archives
2. **Given** monthly archives older than the long-term retention period exist, **When** the cleanup job runs, **Then** those archives are deleted or moved to cold storage
3. **Given** the cleanup job runs, **When** archiving completes, **Then** the repository size is reduced while maintaining data accessibility through archive extraction

---

### User Story 3 - Frontend-Ready Data Aggregation (Priority: P3)

As a frontend developer, I need pre-aggregated summary data files so that the static frontend can load and display electricity trends quickly without processing thousands of individual daily files.

**Why this priority**: Frontend performance depends on having aggregated data readily available. This enables smooth user experience once data is collected.

**Independent Test**: Can be tested by generating aggregated JSON files from the daily data and measuring that the frontend loads the summary data within acceptable time limits (under 1 second).

**Acceptance Scenarios**:

1. **Given** daily data files exist for multiple days, **When** the aggregation job runs, **Then** a summary file is generated with statistics per room: current balance, 7-day consumption average, 30-day trend, minimum balance, maximum balance
2. **Given** aggregated summary files exist, **When** the frontend requests data, **Then** the response size is under 500KB for all rooms combined
3. **Given** new daily data is collected, **When** aggregation runs, **Then** the summary files are updated incrementally without reprocessing all historical data

---

### Edge Cases

- What happens when the university epay website is temporarily unavailable during a scheduled run? (Entire batch is rolled back and retried next day)
- How does the system handle rate limiting or IP blocking from the university server? (Batch rollback and retry)
- What happens if a room ID becomes invalid (e.g., room is demolished or merged)? (Batch fails and rolls back - invalid room must be removed from configuration)
- How does the system behave when the GitHub Actions monthly minutes quota is exhausted?
- What happens if the cookie expires mid-query during a batch run? (Partial results rolled back, previous day's data preserved)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST run the electricity query script automatically once per day at a configurable time
- **FR-002**: System MUST read the list of room IDs from an existing configuration file (generated by `list_room_ids.py` or manually maintained)
- **FR-003**: System MUST store each day's query results as individual JSON files using the existing directory structure in an atomic batch operation (all rooms or none)
- **FR-004**: System MUST rollback all partial results if any room query fails during daily collection, preserving previous day's data
- **FR-005**: System MUST use GitHub Secrets for storing the authentication cookie to prevent credential exposure
- **FR-006**: System MUST log all query attempts including success/failure counts and error details
- **FR-007**: System MUST compress daily JSON files older than 30 days into monthly archives
- **FR-008**: System MUST delete monthly archives older than 365 days (or move to external storage)
- **FR-009**: System MUST generate aggregated summary JSON files for frontend consumption after each successful daily run
- **FR-010**: System MUST validate cookie validity before starting batch queries and fail fast if expired
- **FR-011**: System MUST send notification to repository admin via GitHub Actions automatic email notifications when workflow failures occur
- **FR-012**: System MUST support manual workflow triggers via GitHub Actions UI for ad-hoc data collection

### Key Entities

- **Daily Data File**: Individual JSON file containing electricity data for one room on one specific date, stored in hierarchical path structure `{campus}/{building}/{room}-{id}/{date}.json`
- **Monthly Archive**: Compressed file (e.g., `.tar.gz` or `.zip`) containing all daily JSON files for one month across all rooms, stored in `database/archives/` directory within the repository
- **Aggregated Summary**: JSON file containing pre-computed statistics for all rooms including current electricity balance, 7-day consumption average, 30-day trend, minimum balance, maximum balance, optimized for frontend loading
- **Room Configuration**: List of room IDs to query, stored in a version-controlled configuration file
- **Workflow Run Log**: Record of each automated execution including timestamp, duration, success/failure counts, and error messages

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Daily data collection completes within 30 minutes for up to 500 rooms (using existing 24 concurrent connections)
- **SC-002**: Repository size growth is limited to under 10MB per month after compression and archiving
- **SC-003**: Frontend can load aggregated summary data in under 1 second on standard broadband connection
- **SC-004**: At least 95% of daily scheduled runs succeed without manual intervention (excluding cookie renewal periods)
- **SC-005**: Historical data remains queryable for trend analysis spanning at least 12 months
- **SC-006**: Failed queries are logged with specific error types enabling quick diagnosis and resolution
- **SC-007**: No sensitive credentials (cookies, tokens) are ever committed to the repository in plain text

## Assumptions

- The authentication cookie has a validity period of at least 7 days, allowing manual renewal when failures occur
- GitHub Actions provides sufficient free minutes for daily execution (estimated ~1 minute per run)
- The university epay website maintains rate limits that allow 24 concurrent connections
- Frontend will be a static site that reads from JSON files in the repository or GitHub Pages
- Monthly aggregation strategy balances storage efficiency with frontend performance needs
- Users will monitor GitHub Actions notifications for failed runs and update cookies as needed
